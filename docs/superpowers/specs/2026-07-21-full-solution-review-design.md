# 证明与推导题完整解答流程设计

## 目标

对需要完整推导或证明的数学题，提交可独立判分的中文解答，而不是仅提取最终短答案；普通计算题维持既有的答案提取、投票和最终答案整理流程。

## 适用范围

题干包含“证明、推导、说明理由、解释、论证、求证、验证”任一关键词时，进入完整解答流程。其他题目不改变现有行为。

## 数据流

1. 生成多个候选解答。对完整解答题，接受具有非空内容的候选，即使其未使用“【最终答案】”标记。
2. 审核器比较候选，返回候选编号与简短修订意见。编号无效、格式不符或请求失败时，选择内容最长的候选并记录降级原因。
3. 整理器只基于题目、入选候选和审核意见，输出含“【解答】”和“【结论】”的完整中文解答。
4. 若整理结果为空、过短或缺少必要结构，直接回退到入选候选；审核与整理的原始响应及是否回退均写入 trace。

## 约束与错误处理

- 审核器不重新解题或补造结论；整理器不得引入候选不支持的新结论。
- 所有新增模型调用均使用零温度，并沿用调用方的 client 接口。
- 单个审核或整理调用失败不得导致整题失败，必须降级为可用候选。
- API 密钥只能由 `INTERN_API_KEY` 环境变量提供，不能写入源码、文档、日志或提交记录。

## 测试

- 审核器选择指定候选时，整理器接收该候选与审核意见，最终返回其合格整理结果。
- 审核器返回无效编号时，流程选择内容最长候选。
- 整理结果缺少“【解答】”或“【结论】”时，流程回退入选候选。
- 非完整解答题继续使用原有答案提取路径。
