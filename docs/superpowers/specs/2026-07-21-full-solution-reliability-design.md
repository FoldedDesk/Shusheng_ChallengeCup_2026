# 完整解答链路可靠性设计

## 目标

提高证明、推导和解释类数学题的完整解答质量，同时避免对已有答案明显一致的候选额外发起审核或整理模型调用。

## 范围

本次只调整 `ReasoningAgent` 的完整解答分流、候选质量判断和调用决策，并为这些行为补充离线单元测试。普通数值题的提取、投票和 finalizer 链路保持不变。

## 题型分流

完整解答题由中英文请求词识别，覆盖“证明、推导、解释、论证、求证、验证”以及 `prove`、`derive`、`show all steps`、`explain` 等表达。识别后，候选生成提示要求给出推导和明确结论。

候选必须同时包含实质性解答内容和明确结论，才可进入完整解答分支。仅含 `【最终答案】`、空白、或过短的文本不作为合格候选。若没有合格候选，保留原有的安全错误返回，而不把不完整的文本提交为完整解答。

## 本地质量门与调用策略

每个合格候选经本地规则获得质量分：推导迹象、结论标识、有效文本长度均参与判断。先按规范化的明确结论分组。

当所有合格候选结论一致时，直接返回质量最高的原候选，不调用审核器或整理器。当结论不一致时，才调用审核器选择候选；仅当审核意见指出明确问题时调用整理器。审核或整理失败时，返回本地质量最高的合格候选。

整理结果仍必须有 `【解答】` 与 `【结论】`，并且结论部分包含非空的具体文本；否则丢弃整理结果并回退。

## 可观测性

trace 记录完整题的本地候选筛选、是否因结论一致跳过模型调用、审核选择、是否调用整理器和回退原因。trace 不记录敏感数据。

## 测试

使用顺序假客户端补充离线单测，验证：英文完整题分流；只给最终答案的候选被拒绝；一致候选不会产生审核/整理调用；冲突候选经审核后在需要时整理；无效审核或无效整理均回退到质量最高候选。
