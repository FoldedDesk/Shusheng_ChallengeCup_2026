# 更新日志 FD08.2.0

## 概述

一次版本升级包含四项优化 + 两项修复：

**新增功能（四项）：**
1. LLM 答案提取（替换脆弱的正则提取）
2. Verifier 独立求解 + 答案匹配（替换主观的 VERDICT A/B）
3. 多数投票选择（候选答案一致时跳过 verifier）
4. 分步推理（可选的多阶段生成，默认关闭）

**后续修复（两项）：**
5. 标记化解析——防御 Intern-S Thinking Process 污染
6. Token 配额扩容——修复 policy 生成截断

## 相比 FD08.1.0 的改动

### 新增提示词
- `STAGE_PROMPTS[4]` — 四阶段推理指令（分析 → 思路 → 计算 → 答案）
- `EXTRACTION_PROMPT` — 专用答案提取提示词

### Verifier 重设计
- `VERIFIER_PROMPT` 从"判断是否正确"改为"独立求解 + 比对答案"
- 输出格式：`SELF_ANSWER:` / `MATCH: YES/NO` / `CONFIDENCE:` 标签格式
- `_parse_verdict()` 用三个独立 `findall(...)[-1]` 解析标签（防御 thinking 污染）

### 新增方法
- `_llm_extract_answer()` — 用 `ANSWER:` 标签格式提取，正则容错 NO_ANSWER 变体
- `_regex_extract_answer()` — 正则回退（含尾部标点清理）
- `_generate_candidates_oneshot()` — 原一次性生成
- `_generate_candidates_staged()` — 四阶段顺序生成
- `_generate_candidates()` — 根据 `use_staged_reasoning` 分发

### 新增配置参数
| 参数 | 默认值 | 说明 |
|------|--------|------|
| `use_llm_extraction` | True | LLM 答案提取开关 |
| `extraction_max_tokens` | 1024 | 提取响应最大 token 数（thinking 吃 ~400） |
| `majority_vote_threshold` | 0.5 | 多数共识最低占比 |
| `use_staged_reasoning` | False | 分步推理开关 |
| `staged_stages` | 4 | 推理阶段数 |

### Token 配额
| 参数 | 旧值 (FD08.1.0) | 新值 (FD08.2.0) |
|------|----------------|-----------------|
| `max_tokens` (policy) | 4096 | **8192** |
| `extraction_max_tokens` | 无 | **1024** |

### 最终答案选择流程

整个 `solve()` 方法经过五个阶段，从生成候选推理到选出最终答案：

---

**第一阶段：生成候选推理（Phase 1: Generate）**

由 Policy Agent 调用 Intern-S 模型，对同一道题生成 `policy_sample_times`（默认 3）份独立的推理过程。

有两种生成模式（由 `use_staged_reasoning` 控制，默认 `False`）：

- **一次性模式（默认）**：每次调用直接要求模型"分析题意 → 逐步推导 → 最终答案"。一次 API 调用完成整条推理。
- **分步模式**：将推理拆成四步串行调用：
  1. 分析已知条件和求解目标
  2. 提出解题策略
  3. 逐步计算推导
  4. 输出最终答案（`【最终答案】<答案>`）

分步模式每道题消耗 4 倍 API 调用，暂未启用。一次性模式下，`policy_temperature=0.6` 提供多样性，`max_tokens=8192`（预留 ~400 token 给 Intern-S thinking）。

---

**第二阶段：答案提取（Phase 2: Extract）**

对每份候选推理，从文本中提取出纯答案值。采用 **LLM 优先 + 正则回退** 策略：

1. **LLM 提取**：调用 Extraction Agent（temperature=0，max_tokens=1024），输入候选推理全文，要求按 `ANSWER: <答案>` 格式输出。
   - Intern-S 必定先输出 Thinking Process，再输出正式回复。解析时用 `re.findall(r"ANSWER\s*[:：]\s*(.+?)(?:\n|$)", text, re.I)[-1]` 取**最后一个** `ANSWER:` 行，自动跳过 thinking。
   - 提取后做两步清洗：
     - **NO_ANSWER 检测**：用正则 `^NO[_\-\s]*ANSWER` 匹配变体（`NO_ANSWER`、`NO-ANSWER`、`NO_ANSWER.` 等），命中则丢弃。
     - **尾部标点清理**：`re.sub(r"[\`'\".,;:!?）\]】\s]+$", "", ans)` — Intern-S 常在答案后加 backtick、句号等。

2. **正则回退**（LLM 提取失败时）：在**原始候选文本**上按优先级依次匹配：
   - `【最终答案】<答案>` 标签
   - `\boxed{<答案>}` LaTeX 格式
   - `最终答案：<答案>` / `答案：<答案>`
   - 行末 `= <value>`（限 ≤30 字符且含数字，防止匹配整句）
   - 末行以数字开头且非步骤标签

提取结果写入 trace（含 `method`、`answer`、LLM `raw_response`），同时对所有答案做归一化（去 `$`、压缩空白、标点统一），供后续投票比较。

---

**第三阶段：多数投票（Phase 3: Majority Vote）**

统计归一化后的答案分布。若**同一答案出现 ≥2 次**且**占比 ≥ `majority_vote_threshold`（默认 0.5）**：

→ 直接采纳该答案为最终结果，**跳过第四阶段验证器**。节省 2N 次 API 调用（N 个候选 × 2 次验证）。

若达不到多数共识 → 进入第四阶段。

---

**第四阶段：验证器评分（Phase 4: Verify & Score）**

对每个候选，调用 Verifier Agent（temperature=0，max_tokens=1024）进行 `verifier_voting_times`（默认 2）次独立验证。

**验证流程**：
1. 输入：原题 + 候选推理全文
2. Verifier 独立求解题目，输出三标签格式：
   ```
   SELF_ANSWER: <自己算出的答案>
   MATCH: YES 或 NO
   CONFIDENCE: <0-10>
   ```
3. 解析（`_parse_verdict`）：对 `SELF_ANSWER:`、`MATCH:`、`CONFIDENCE:` 分别 `findall(...)[-1]` 取最后匹配（防御 thinking），得到 `(is_match, confidence, self_answer)`。
4. 单次验证分数：`confidence / 10`（MATCH=YES 时），否则 0.0。

**候选评分公式**：
```
verifier_score = avg(各次验证的 confidence/10)
consistency_bonus = (相同答案的候选数 - 1) × 0.15
total_score = verifier_score + consistency_bonus
```

- `verifier_score`：Verifier 独立求解后是否认可该候选答案（0~1）
- `consistency_bonus`：其他候选也得出同样答案时给予奖励（0~0.3），
  鼓励"被多个独立推理支持的答案"

最终选择 `total_score` 最高的候选。

---

**第五阶段：返回结果（Phase 5: Return）**

```python
return {
    "final_response": extracted_answers[best_candidate_id],
    "trace": trace,  # 完整推理过程，含所有阶段中间结果
}
```

`final_response` 是纯答案值（如 `"72"`、`"-1"`、`"-1/8"`），`trace` 记录完整过程供分析调试。

---

**三种选择路径的概率分布（dev 集实际表现）**：

| 路径 | 条件 | 触发频率 | 可靠性 |
|------|------|---------|--------|
| 多数投票 | ≥2/3 候选答案一致 | 高（三题全中） | ★★★★ |
| 验证器评分 | 无共识时 | 低（暂未触发） | ★★★ |
| 正则回退 | LLM 提取失败 | 低（~1/3 调用） | ★★☆ |

### Trace 变更
- 新增 `extract_answer_{i}` 步骤，含 `method`、`answer`、`raw_response`
- verifier 调用包含 `self_answer`、`parsed_match`、`parsed_confidence`
- 多数投票成功时新增 `majority_vote` 步骤
